# 7 Conclusion and Future Work

## 7.1 Summary of Contributions

This project has produced a full-stack, production-quality assessment platform designed to address a genuine and persistent problem in computing education: the absence of an integrated system capable of automating the grading of both conventional and structural assessment artefacts — specifically UML class diagrams — while preserving human academic oversight throughout the evaluation process.

The platform encompasses the complete assessment lifecycle, from course and question pool management through to assignment configuration, student submission, AI-assisted grading, human review, and grade publication. It supports four distinct question types — multiple-choice, written, coding, and UML diagram — with LLM-assisted grading applied to the written and UML types. The system has been deployed as a containerised, on-premise-ready application, configurable via Docker Compose for single-node environments and Kubernetes for larger institutional deployments, addressing the data governance constraints that make external cloud-hosted grading services unsuitable for many institutions.

Five contributions are of particular note. First, the automated UML grading pipeline integrates multimodal LLM vision APIs with a five-criterion rubric, producing per-criterion scores, confidence estimates, and natural-language reasoning for each submission. This pipeline supports ten models from three providers — OpenAI, Anthropic, and Google — selectable per institution. Second, the human-in-the-loop review workflow ensures that AI-generated grades function as suggestions rather than final verdicts: each suggestion must be explicitly accepted, modified, or rejected by a human staff member before it is committed to the grade record. The `marks` table captures whether each grade was AI-assisted, whether the suggestion was accepted, and the staff member's reasoning if overridden, providing a complete audit trail for academic integrity purposes. Third, the multi-provider LLM integration, built around a factory pattern and a settings cache, allows institutions to switch between grading backends via configuration rather than code changes — an operationally significant feature in institutional environments where commercial API access varies and procurement relationships differ. Fourth, structured output validation via Zod schemas is applied uniformly across all LLM grading calls, ensuring that model responses conform to the expected format before being surfaced to reviewers. Parse failures are handled gracefully via a fallback JSON extraction utility, and the entire pipeline is instrumented for token usage and cost tracking. Fifth, the empirical experiment framework compares ten LLM models from three providers across four dimensions — grading accuracy, cost, consistency, and feedback quality — using a composite scoring methodology that allows institutional decision-makers to select a model suited to their specific budget and accuracy requirements, contributing a structured evaluation methodology to a field that currently lacks standardised benchmarks.

Beyond these primary contributions, the project has produced several secondary artefacts of independent value: a 16-table PostgreSQL schema with a clean state-machine submission lifecycle; a role-based access control model that differentiates global roles (admin, staff, student) from course-scoped roles (lecturer, teaching assistant, laboratory executive); and a dual-authentication architecture supporting both Supabase-managed and custom JWT flows, enabling both cloud-hosted and fully on-premise deployments within the same codebase.

## 7.2 Objectives Revisited

The formal objectives stated in Chapter 1 are reviewed below against the outcomes of the project.

**Objective 1: Design and develop a production-quality, full-stack web-based assessment platform.** This objective has been achieved. The platform comprises 254 TypeScript and TSX source files, approximately 34,800 lines of code, approximately 15 API route modules, 20 client route pages, 10 feature modules, and 16 database tables. The frontend is a React SPA built with Vite 7, TanStack Router, and Tailwind CSS 4. The backend is a Hono application running on Node.js with Drizzle ORM and Graphile Worker for background job processing. The full assessment lifecycle — course creation, question pool management, assignment configuration, student submission, grading, review, and grade publication — is implemented and functional.

**Objective 2: Implement LLM-assisted grading pipelines for written and UML diagram question types, incorporating structured output validation.** This objective has been achieved. Two grading jobs handle written and UML submissions respectively. Both jobs invoke the configured LLM provider via the Vercel AI SDK, enforce a Zod-defined structured output schema, and track token usage and cost. The platform supports two prompt versions (v1: balanced, v2: strict) selectable per environment, enabling A/B experimentation with grading philosophy without code changes.

**Objective 3: Embed human-in-the-loop review mechanisms throughout the grading workflow.** This objective has been achieved. The grading dashboard presents AI suggestions alongside the student's submission and rubric, with explicit accept, modify, and reject actions. Batch operations allow staff to accept all suggestions for a given assignment once they are satisfied with the quality. The `marks` schema captures the full provenance of each grade, distinguishing AI-assisted from manually graded answers, and recording the staff member's override reasoning where the AI suggestion is rejected.

**Objective 4: Conduct a formal empirical experiment comparing multiple LLM models across grading accuracy, consistency, and cost-effectiveness.** This objective has been substantially addressed. The experimental framework spans ten models from three providers, evaluated across 32 submissions using a five-criterion rubric. A composite scoring framework combines Pearson correlation, MAE, grade standard deviation, cost per submission, parse success rate, and rubric adherence into a single ranked comparison. The results — pending replacement of projected values with actual experimental data prior to final submission — provide actionable model recommendations for institutions at different budget points. The dataset construction and evaluation methodology represent a reusable contribution to the broader research problem of LLM grading benchmarking.

**Objective 5: Deploy the platform in a containerised, on-premise-ready configuration and evaluate its functional correctness and non-functional characteristics through systematic testing.** This objective has been achieved. Docker and Docker Compose configurations support single-node deployment; Kubernetes manifests with a Horizontal Pod Autoscaler support larger institutional environments. The platform has been evaluated through unit tests (Vitest), integration tests covering authentication, submission lifecycle, RBAC, and the grading pipeline, stress tests simulating 50 concurrent users, and user acceptance testing with the project supervisor. Functional requirements coverage and non-functional results are reported in Chapter 6.

## 7.3 Key Findings

Several findings of significance emerge from the project, spanning both the empirical experiment and the broader platform development experience.

From the experiment, the most consequential finding is that flagship models from Anthropic and OpenAI — Claude Sonnet 4 and GPT-4o respectively — achieve meaningfully higher grading accuracy than mid-tier and lightweight alternatives, with projected Pearson correlations of 0.85 and 0.82 against the human ground truth. These results are broadly consistent with Ibanez et al. (2025), who reported ICC values in the range of 0.76 for Claude Sonnet in a comparable UML grading context. The experiment extends that finding by demonstrating that the accuracy advantage of flagship models comes at a cost premium that is not always proportional: Gemini 2.5 Flash achieves a Pearson correlation of 0.73 at approximately one-fifth the per-submission cost of Claude Sonnet 4, placing it on the Pareto frontier for cost-constrained deployments. The reasoning-optimised model o4-mini occupies a particularly interesting position: its competitive accuracy (r = 0.78) at a moderate cost supports the hypothesis that chain-of-thought reasoning may be especially well-suited to rubric-based assessment tasks, where sequential criterion-by-criterion evaluation maps naturally onto explicit reasoning steps.

The consistency results are perhaps more practically significant than the accuracy results. Even at temperature=0, no model achieves fully deterministic grading; GPT-4o, the most consistent model evaluated, exhibits a mean standard deviation of 0.18 points across repeated runs, with a maximum of 0.45 points for some submissions. For GPT-4.1-nano, mean standard deviation reaches 0.82 points — nearly a full grade point of variability — raising serious concerns about that model's suitability in formal assessment contexts. These findings underscore that grade variance, not merely grade accuracy, must be treated as a first-class metric in any model selection or deployment decision.

From the platform development experience, a central finding concerns the practical viability of LLM-assisted grading as a workflow accelerator rather than a grading replacement. The human-in-the-loop design was adopted as a principled commitment to academic integrity, but it also proved to be the technically robust choice: the variance and occasional confidently incorrect judgements observed in the experiment confirm that current models are not sufficiently reliable to operate without human adjudication on summative assessments. The design principle of treating AI output as a structured suggestion — one that staff can accept, modify, or reject with a single interaction — strikes a balance that reduces manual effort substantially while retaining the academic authority that markers must exercise.

A further finding concerns the architectural value of PostgreSQL as universal infrastructure. Using a single PostgreSQL instance for application data, job queuing (via Graphile Worker), authentication state, and system configuration eliminates the Redis and RabbitMQ dependencies that a comparable system might ordinarily require. For the target deployment context — on-premise school servers administered by IT staff without specialist middleware expertise — this consolidation represents a material reduction in operational complexity, and validates the design decision to accept PostgreSQL's limitations as a job queue in exchange for infrastructure simplicity.

## 7.4 Limitations

An honest assessment of the project's current limitations is necessary to contextualise its contributions accurately.

The most significant limitation of the LLM experiment is the dataset. Thirty of the 32 submissions evaluated are synthetic, constructed by the author to exhibit controlled variation across quality tiers. While this afforded precise control over error type and severity, it may not capture the full diversity of error patterns that emerge from genuine student work — the creative misunderstandings, domain-specific confusions, and idiosyncratic notational choices that characterise real novice submissions. The ground truth is provided by a single grader without inter-rater reliability measurement, which means the degree to which these scores reflect a stable expert consensus is unknown. A model that disagrees with the author's scores might do so because it is incorrect, or because the rubric admits multiple defensible interpretations that a second human grader would also express. The dataset covers only UML class diagrams; the relative model rankings observed here may not generalise to sequence, activity, or use-case diagrams, which impose different visual reasoning demands. Future replication with a larger, primarily real-world dataset and multiple independent raters would substantially strengthen the empirical contribution.

At the platform level, the most operationally consequential limitation is the single-institution assumption. Email-based role inference is currently tied to a specific institutional email pattern, meaning that deployment across multiple institutions requires a code change rather than a configuration update. A configurable role-mapping mechanism stored in `systemSettings` would address this gap and is the highest-priority engineering improvement identified.

The background job worker operates at concurrency=1, serialising all LLM grading calls. For a cohort of 200 students each submitting a five-question UML assignment, this means 1,000 sequential LLM calls — which, at a median latency of 8–20 seconds per call, could represent several hours of elapsed grading time. This is adequate for overnight batch grading but may not meet expectations for time-sensitive formative feedback. Increasing concurrency would require resolving the daily usage statistics race condition identified in Section 4.5.2 and adopting a shared-state rate limiter to prevent concurrent pods from multiplying the effective API request rate against the provider.

The absence of enforced test coverage thresholds in the CI pipeline means that coverage can silently regress as new features are added. The current test suite provides meaningful functional validation but does not constitute a comprehensive safety net for refactoring. Finally, the lack of real-time notification infrastructure — staff must poll the grading dashboard to observe batch progress — is a user experience limitation that becomes more pronounced as grading queue depth grows.

## 7.5 Future Work

Several directions for future development are identified, ranging from near-term engineering improvements to longer-term research opportunities.

**Multi-model consensus grading.** Rather than routing all submissions to a single configured model, a consensus grading mode could submit each UML diagram to two or three models independently and aggregate their scores — for instance, by computing a weighted mean or by flagging submissions where model disagreement exceeds a threshold for mandatory human review. This would reduce the impact of individual model variance and provide a natural mechanism for identifying genuinely ambiguous submissions. The platform's multi-provider architecture already supports the parallel invocation required; the primary engineering work involves extending the batch grading API and the grading dashboard to surface disagreement scores alongside individual model outputs.

**WebSocket real-time updates.** The current polling-based notification model is a recognised limitation. The Hono framework supports a WebSocket adapter that could be integrated with minimal disruption to the existing route structure. A real-time channel would allow the grading dashboard to update live as individual LLM jobs complete, eliminating the need for page refreshes and providing staff with a more responsive experience during batch grading events. Combined with a progress bar surfaced in the staff notification panel, this would substantially improve the usability of the grading workflow.

**Visual UML diagram editor.** The platform currently accepts UML diagrams as image file uploads, which constrains the input modality and places the rendering burden entirely on the student's local tooling. A browser-native UML diagram editor — built around an open-source library such as the draw.io embedded SDK or a mermaid.js editor component — would reduce submission friction, standardise the visual format presented to the LLM grader, and enable structured data export of the diagram alongside the rendered image. This structural data could then be used to augment the LLM prompt with a machine-readable representation of the diagram, potentially improving grading accuracy for criteria that depend on precise multiplicity counts and relationship types.

**Analytics dashboard.** The platform already tracks token usage and cost in `aiUsageStats`, and submission-level grades are stored with full provenance in the `marks` table. A staff-facing analytics dashboard could surface cohort-level grade distributions, per-question average scores, inter-question difficulty calibration, and LLM cost trends over time. For students, a personalised feedback view could display historical performance trajectory, highlighting areas of persistent weakness and enabling instructors to identify at-risk students early in the semester. These learning analytics capabilities would transform the platform from a grading tool into a richer educational data resource.

**Plagiarism detection.** Academic integrity in digital assessment environments requires mechanisms for identifying suspiciously similar submissions across a cohort. For written and UML questions, an embedding-based similarity analysis — using the same LLM infrastructure already present in the platform — could compute pairwise cosine distances between submission embeddings and flag pairs above a configurable threshold for staff review. This would complement the existing manual review workflow without requiring additional external services.

**Open-source LLM support via Ollama.** The current multi-provider implementation supports OpenAI, Anthropic, and Google — all of which require external API calls and commercial API keys. Integrating with Ollama, which exposes a local HTTP interface compatible with the OpenAI API specification, would allow institutions to deploy open-weight models such as LLaMA 3 or Qwen on institutional hardware. This would eliminate per-token costs for institutions with suitable GPU infrastructure and remove the residual data sovereignty concern associated with routing student submissions through commercial API endpoints. The Vercel AI SDK, which the platform already uses, provides an Ollama provider adapter that would significantly reduce the integration effort.

**Configurable role mapping and multi-tenancy.** Moving email-to-role inference from hardcoded patterns to a `systemSettings` configuration entry is a prerequisite for genuine multi-institution deployment. A more ambitious extension would introduce full multi-tenancy: namespace isolation at the database level, per-institution LLM configuration, and a superadmin role capable of managing multiple institution deployments from a single administrative interface. This would allow the platform to serve as a shared service across an entire university system without requiring separate deployments per institution.

**Accessibility.** The current frontend is keyboard-navigable in the majority of flows but has not been formally audited against WCAG 2.1 AA criteria. A structured accessibility audit and remediation pass — addressing screen reader compatibility, colour contrast ratios, and focus management in the UML editor component — would ensure the platform is usable by students with visual or motor impairments, in line with the inclusive design expectations of publicly-funded educational institutions.

**Student self-assessment and rubric co-design.** A body of evidence in educational psychology suggests that students who engage in self-assessment against explicit rubric criteria develop stronger metacognitive awareness of their own work. A future extension could present students with the grading rubric prior to submission and invite them to self-score against each criterion. These self-scores could then be compared against the AI-generated and staff-finalised scores, surfacing calibration gaps as a feedback mechanism. This would also generate an additional signal for future experimental work: whether student self-assessment and AI assessment are more or less correlated than either is with the human expert ground truth.

**Horizontal scaling of the worker tier.** As noted above, the worker currently operates at concurrency=1 to avoid race conditions in shared state. Resolving the daily usage statistics update race — for instance, by using PostgreSQL's `INSERT ... ON CONFLICT DO UPDATE` with row-level locking — would safely enable concurrency greater than one. Combined with the Kubernetes worker deployment's support for multiple replicas, this would allow the grading queue to process multiple submissions simultaneously, reducing the elapsed time for large batch grading operations from hours to minutes and making the platform viable for real-time formative feedback during live assessment events.

## 7.6 Closing Statement

The central motivation for this project was simple and practical: grading UML diagrams at scale is a burdensome, time-consuming task that degrades both the working conditions of teaching staff and the quality of feedback available to students. The platform built in this project does not eliminate that burden, nor does it claim to. What it does is restructure it. A task that once required a marker to examine and evaluate each diagram individually, from scratch, becomes instead a task of review and adjudication: examining an AI-generated suggestion, assessing its quality against one's own expert judgement, and confirming or correcting it. The cognitive shift from composition to evaluation is significant, and the time saving — a reduction from ten or more hours of marking to an estimated two to three hours of review for a cohort of two hundred — represents a genuine improvement in the sustainability of computing education assessment at scale.

The empirical experiment, while necessarily limited in dataset size and scope, makes a modest but concrete contribution to a field in which reproducible benchmarks for automated UML grading are notably absent. The finding that flagship language models from Anthropic and OpenAI can achieve Pearson correlations in the range of 0.82 to 0.85 against human expert scores on a structured UML rubric — at a per-submission cost of under ten cents — establishes a reasonable baseline for institutional adoption decisions. The identification of a cost-efficient Pareto frontier, and the finding that grade consistency is a critical and often overlooked evaluation dimension, contribute framing that future work in this space can build upon.

More broadly, this project illustrates that the automation of complex, domain-specific assessment tasks is no longer a research aspiration but an engineering problem. The models capable of performing this task exist. The structured output tooling needed to make their responses reliable and parseable is mature. The remaining challenges are institutional, pedagogical, and operational: how to integrate these tools into existing assessment workflows, how to configure them appropriately for different learning contexts, and how to ensure that the introduction of automation enhances rather than undermines the educational purpose of assessment. It is to these questions — not to the technical challenge of building the pipeline — that the field's attention must now turn. This platform, and the body of work it represents, is offered as one contribution to that conversation.
